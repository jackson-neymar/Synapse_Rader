# Synapse_Rader 项目学习与面试讲解指南

这份文档的目标不是让你背代码，而是让你能把项目讲清楚、讲专业、讲得像自己真的做过工程取舍。

读完后，你应该能回答三类问题：

- 这个项目解决了什么问题，整体架构是什么？
- 每个核心模块为什么这么设计，技术难点在哪里？
- 如果面试官追问缺陷、扩展性、稳定性，你怎么回答？

---

## 1. 先用一句话讲清楚项目

Synapse_Rader 是一个基于 FastAPI、LangGraph 和 LLM 的 AI 技术情报自动监控系统，可以从 GitHub、HuggingFace、Hacker News、arXiv 等多个信源自动采集 AI 技术动态，经过关键词过滤、LLM 分类去噪、RAG 历史对比和四维价值评分后，生成结构化日报，并通过 Web 控制台和飞书推送给团队。

面试时可以这样说：

> 我做的是一个 AI 技术情报自动化系统。它每天自动从 GitHub、HuggingFace、Hacker News、arXiv 等渠道抓取 AI 相关信息，然后通过多智能体流水线完成筛选、去噪、RAG 历史对比、LLM 评分和日报生成，最后在 Web 控制台展示并推送到飞书。项目重点不是单纯调用大模型，而是把采集、筛选、分析、存储、检索和分发做成一个完整闭环。

这段话面试官通常会听出三个亮点：

- 你做的是完整系统，不是简单 demo。
- 你用了 Agent 编排、RAG、异步采集、结构化评分。
- 你理解业务价值：把海量信息变成可决策的技术情报。

## 2. 项目背景：它解决了什么痛点？

AI 领域更新非常快，GitHub 每天都有新项目，arXiv 每天都有论文，HuggingFace 上模型频繁更新，Hacker News 上也有很多技术讨论。

如果人工每天跟踪这些信息，会遇到几个问题：

- 信息源分散，需要反复切换平台。
- 噪声很高，大量内容只是营销、重复发布或低价值讨论。
- 缺少统一评价标准，不知道哪些内容值得团队关注。
- 历史信息难以关联，无法判断某个技术是新趋势还是旧热点重复出现。
- 团队分享成本高，人工整理日报耗时。

所以这个项目把流程自动化：

```text
多源采集 -> 去重 -> 关键词初筛 -> LLM 分类 -> LLM 去噪
        -> RAG 历史对比 -> 四维评分 -> 日报生成 -> Web 展示 / 飞书推送
```

面试表达：

> 这个项目的核心价值是把“人肉刷信息”变成“自动化情报流水线”。系统不是把所有抓到的信息都丢给用户，而是先筛选、再分析、再分级推荐，最后输出可以直接用于团队决策的日报。

## 3. 技术栈怎么讲？

项目技术栈可以按层讲，不要一口气报一串名词。

| 层级 | 技术 | 作用 |
| --- | --- | --- |
| 后端 API | FastAPI | 提供日报、检索、手动触发、运行状态接口 |
| Agent 编排 | LangGraph | 把采集、筛选、分析、报告、分发编排成流水线 |
| LLM 调用 | LangChain + DeepSeek 兼容接口 | 分类、去噪、评分、总结 |
| 数据采集 | httpx + feedparser | 异步抓取 GitHub、HF、HN、arXiv 数据 |
| 关系数据库 | SQLAlchemy Async + SQLite | 保存原始条目、分析结果、日报和执行日志 |
| RAG 检索 | ChromaDB + Embedding + BM25 | 检索历史相似情报，辅助分析 |
| 调度 | APScheduler | 定时执行日报任务 |
| 前端 | Vue 3 + TypeScript + Naive UI | Web 控制台 |
| 分发 | 飞书开放平台 | 群消息和文档推送 |

面试表达：

> 后端用 FastAPI 承接 API，核心任务流用 LangGraph 编排成五个节点。数据层用 SQLAlchemy Async 管 SQLite，向量检索用 ChromaDB。采集端用 httpx 异步请求多个外部信源，分析端通过 LangChain 调 DeepSeek 兼容接口。前端用 Vue 3 和 Naive UI 做控制台。

## 5. 核心架构：五个 Agent 各自做什么？

整个项目最重要的架构是 LangGraph 五节点流水线：

```text
Collector -> Curator -> Analyst -> Editor -> Dispatcher
```

### 5.1 Collector：采集智能体

对应文件：

- `backend/graph/nodes/collector.py`
- `backend/graph/tools/fetchers.py`
- `backend/graph/tools/storage.py`

它负责从多个信源抓取数据：

- GitHub 新项目和热门项目
- HuggingFace 模型更新
- Hacker News AI 相关讨论
- arXiv AI/NLP/CV/ML 论文

关键设计：

- 使用 `asyncio.gather` 并发执行多个 fetcher。
- 每个 fetcher 单独异常处理，一个信源失败不影响其他信源。
- 使用 `SHA256(url + title)` 生成稳定 ID 做去重。
- 新数据写入 `raw_items` 表。

面试表达：

> Collector 主要解决多源采集和容错问题。我用异步并发同时请求多个外部 API，这样整体耗时取决于最慢的信源，而不是所有信源耗时相加。同时每个信源失败只记录错误，不中断整个流程。采集后基于 URL 和标题生成 SHA256 ID 做去重，避免同一条信息重复进入后续 LLM 分析。

常见追问：

**为什么要异步采集？**

因为外部 API 请求主要是 I/O 等待，异步可以提高吞吐，避免一个接口慢导致整个流程阻塞。

**为什么用 URL + title 去重？**

URL 能定位原始内容，title 能进一步区分同源页面或镜像内容。组合后做 hash，可以得到长度固定、适合做主键的 ID。

---

### 5.2 Curator：筛选智能体

对应文件：

- `backend/graph/nodes/curator.py`
- `backend/config/keywords.py`
- `config/keywords.yaml`

Curator 做三级筛选：

```text
L1 关键词过滤 -> L2 LLM 分类 -> L3 LLM 去噪
```

L1 关键词过滤：

- 成本低，速度快。
- 用关键词先过滤明显无关内容。
- 英文关键词做大小写不敏感匹配，中文关键词做子串匹配。

L2 LLM 分类：

- 判断内容是否属于目标领域，比如 AI Agent、多模态大模型、NLP。
- 输出一级分类和二级标签。

L3 LLM 去噪：

- 去掉营销内容、灌水内容、重复无价值内容。

面试表达：

> Curator 我设计成三级筛选，而不是直接把所有内容丢给 LLM。第一层关键词过滤成本最低，先快速去掉明显无关内容；第二层用 LLM 判断领域归属；第三层用 LLM 做内容质量判断，去掉营销和噪声。这样可以减少 LLM 调用成本，也能提升后续分析质量。

常见追问：

**为什么不用 LLM 一步完成所有筛选？**

因为直接把所有原始数据丢给 LLM 成本高、延迟大，而且无关数据会影响模型判断。先规则过滤，再 LLM 精筛，是成本和效果之间的折中。

**如果 LLM 输出不是合法 JSON 怎么办？**

当前代码会尝试去掉 Markdown 代码块并提取 JSON 区间，解析失败时选择降级策略，比如分类失败时放行全部条目，避免流程直接中断。生产环境可以进一步加强为 schema 校验、重试、结构化输出或函数调用。

---

### 5.3 Analyst：分析智能体

对应文件：

- `backend/graph/nodes/analyst.py`
- `backend/graph/tools/prompts.py`
- `backend/graph/tools/rag.py`
- `backend/graph/tools/llm.py`

Analyst 是项目最有技术含量的部分，做三件事：

```text
历史相似情报检索 -> LLM 四维评分 -> 写入数据库和向量库
```

四维评分体系：

| 维度 | 含义 |
| --- | --- |
| 商业潜力 | 有没有真实应用场景、市场热度和业务价值 |
| 落地难度 | 是否开源、文档是否完整、部署成本高不高 |
| 性能指标 | 是否有 benchmark、是否接近或达到 SOTA |
| 业务兼容性 | 是否适合团队现有技术栈和业务方向 |

推荐等级：

- `>= 4.0`：强烈推荐
- `3.0 - 3.9`：值得关注
- `2.0 - 2.9`：暂不跟进
- `< 2.0`：不推荐

面试表达：

> Analyst 会先用当前条目的标题和描述去向量库里检索历史相似情报，然后把这些历史上下文和当前条目一起放进 Prompt，让 LLM 做四维评分。这样模型不是孤立评价一条新闻，而是可以参考历史相似内容，判断它是不是趋势延续、重复热点，或者真的有新增价值。

常见追问：

**RAG 在这个项目里解决什么问题？**

它解决“历史上下文缺失”的问题。比如今天出现一个新框架，单看标题可能觉得很新，但如果历史里已经有类似项目，系统就能提示相似情报，帮助判断它是技术演进还是重复包装。

**为什么要混合检索，而不是只用向量检索？**

向量检索擅长语义相似，但可能忽略关键词精确匹配；BM25 对关键词敏感，适合技术名词、项目名、模型名。混合检索可以兼顾语义相似和关键词命中。

**为什么分析时要并发？**

每条情报的 LLM 分析互相独立，适合并发执行。代码里用 semaphore 限制并发数量，避免瞬间请求过多触发限流。

---

### 5.4 Editor：报告生成智能体

对应文件：

- `backend/graph/nodes/editor.py`

Editor 根据分析结果生成 Markdown 日报。

日报结构：

- 今日概览
- 强烈推荐
- 值得关注
- 暂不跟进
- 分数、置信度、理由、原始链接

面试表达：

> Editor 的职责是把结构化分析结果转成适合人阅读的日报。它会按推荐等级分组，高分内容展示更完整的评分明细和理由，低分内容用更简洁的表格展示。这样用户可以先看重点，再决定是否展开细节。

常见追问：

**为什么报告用 Markdown？**

Markdown 简单、可读、可存储，也方便在 Web、飞书文档或其他平台渲染，是日报类内容比较合适的中间格式。

---

### 5.5 Dispatcher：分发智能体

对应文件：

- `backend/graph/nodes/dispatcher.py`
- `backend/graph/tools/feishu.py`

Dispatcher 做两件事：

- 把日报保存到 `daily_reports` 表。
- 如果配置了飞书凭证，就推送群消息并创建飞书文档。

面试表达：

> Dispatcher 里我把“保存日报”和“外部推送”分开处理。保存数据库是核心路径，必须先成功；飞书是外部依赖，如果没配置或发送失败，系统会记录错误或返回空结果，但不影响日报本身生成。

常见追问：

**外部推送失败怎么办？**

核心数据先落库，外部推送失败不应该导致整个分析结果丢失。后续可以增加重试队列、失败告警和补偿任务。

## 6. 数据库模型怎么讲？

对应文件：

- `backend/models/database.py`

项目有四张核心表。

### 6.1 raw_items

保存原始采集数据。

关键字段：

- `id`：基于 URL + title 的 hash。
- `source`：来源，比如 github、arxiv。
- `title`、`url`、`description`：原始内容。
- `status`：筛选状态。
- `category_l1`、`category_l2`：分类结果。

### 6.2 analyzed_items

保存 LLM 分析结果。

关键字段：

- `raw_item_id`：关联原始条目。
- `summary_one_liner`：一句话总结。
- `score_business`、`score_deploy`、`score_performance`、`score_compatibility`：四维评分。
- `score_total`：总分。
- `recommend_level`：推荐等级。
- `rag_context_used`：使用过的历史上下文。

### 6.3 daily_reports

保存日报。

关键字段：

- `report_date`：日报日期。
- `report_markdown`：Markdown 内容。
- `recommend_high`、`recommend_mid`、`recommend_low`：分级数量。
- `feishu_msg_id`、`feishu_doc_url`：飞书分发结果。

### 6.4 execution_logs

保存执行日志。

关键字段：

- `run_id`：一次运行的 ID。
- `trigger`：scheduled 或 manual。
- `node_name`：执行节点。
- `status`：running、success、failed。
- `error_message`：错误信息。

面试表达：

> 数据库设计上，我把原始采集、分析结果、日报结果、执行日志拆成四张表。这样原始数据和 LLM 分析数据解耦，日报可以按日期归档，执行日志可以支持前端查询任务状态和历史运行记录。

## 8. LangGraph 为什么适合这个项目？

这是一个很容易被问到的问题。

不要只说“因为用了 Agent”，要说清楚为什么需要图编排。

这个项目天然是多阶段流水线：

```text
采集结果 -> 筛选结果 -> 分析结果 -> 报告结果 -> 分发结果
```

每一步都依赖上一步结果，而且每一步职责不同。

LangGraph 的价值：

- 用 `State` 管理节点之间共享数据。
- 每个节点职责清晰，方便调试和扩展。
- 后续可以扩展条件分支，比如没有高价值情报就跳过飞书推送。
- 可以支持 checkpoint 或更复杂的 Agent 工作流。

面试表达：

> 这个项目不是简单的一次 LLM 调用，而是有明确阶段的工作流。LangGraph 比普通函数调用更适合表达这种状态流转，每个节点只关心自己的输入输出，后续如果要增加条件分支、失败重试或人工审核节点，也更容易扩展。

追问：**不用 LangGraph，直接写函数串起来行不行？**

可以。对于 MVP，函数串联也能跑。但 LangGraph 的优势在于流程显式、状态清晰、节点可扩展。如果后续要做条件分支、checkpoint、节点级监控、多 Agent 协作，LangGraph 更合适。

## 9. RAG 为什么不是“为了用而用”？

很多项目简历写 RAG，但面试官会怀疑是不是套概念。这里要讲业务价值。

在 Synapse_Rader 里，RAG 用在 Analyst 阶段：

- 当前条目作为 query。
- 去 ChromaDB 里查历史相似分析条目。
- 把相似历史内容放进 Prompt。
- LLM 基于当前内容和历史上下文一起评分。

它解决的问题：

- 判断是否是重复热点。
- 识别技术演进关系。
- 给日报提供历史对比视角。
- 减少 LLM 对单条信息的孤立判断。

面试表达：

> RAG 在这里不是问答场景，而是用于“历史情报对比”。系统每天都会产生新的分析结果，同时写入向量库。下一次分析新条目时，可以检索相似历史情报，让 LLM 判断这条内容和过去有什么关系。这样日报不只是新闻摘要，而是能体现技术趋势和演进。

## 10. 这个项目的工程亮点

面试时可以重点强调 4 个亮点。

### 亮点 1：完整自动化闭环

不是单点功能，而是从数据源到推送的全链路。

表达：

> 项目从采集、筛选、分析、存储、检索、报告、推送到前端展示形成了闭环，比较接近真实业务系统，而不是单纯的模型调用 demo。

### 亮点 2：规则 + LLM 的分层筛选

降低成本，提高质量。

表达：

> 我没有直接让 LLM 处理所有原始信息，而是先用规则过滤低成本筛掉明显无关内容，再用 LLM 做需要语义判断的分类和去噪，这样在成本、延迟和效果之间做了平衡。

### 亮点 3：RAG 历史对比

让系统有“记忆”。

表达：

> 每条分析结果都会进入向量库，后续分析可以检索历史相似内容。这样系统不只是当天日报生成器，还能逐渐积累团队自己的技术情报记忆。

### 亮点 4：四维评分体系

让 LLM 输出可比较、可解释。

表达：

> 我没有只让模型生成主观总结，而是设计了四维评分和推荐等级，让不同情报之间可以排序和比较。每个分数还附带置信度和理由，提升可解释性。

## 11. 面试官可能追问的缺陷与优化

会讲缺陷并不可怕，反而说明你真的理解项目。

### 11.1 当前节点级进度还不够细

现状：

手动触发接口会记录 pipeline 级日志，但五个 Agent 节点的实时状态还可以进一步细化。

怎么回答：

## 13. 3 分钟项目介绍模板

如果面试官说“详细讲讲”，用这个结构。

> 这个项目的背景是 AI 技术动态非常分散，GitHub、HuggingFace、arXiv、Hacker News 每天都有大量信息，人工跟踪效率很低，所以我做了一个自动化技术情报系统。
>
> 架构上，后端用 FastAPI 提供 API，用 LangGraph 编排核心工作流。工作流分成五个节点：Collector 负责多源异步采集；Curator 负责关键词过滤、LLM 分类和去噪；Analyst 负责 RAG 历史检索和四维评分；Editor 负责生成 Markdown 日报；Dispatcher 负责落库和飞书推送。
>
> 在采集层，我用 httpx 异步请求多个外部信源，并且对每个信源单独做异常处理，保证某个源失败不会影响整个流程。采集后的数据会基于 URL 和标题生成 hash 去重，然后写入 raw_items 表。
>
> 在筛选和分析层，我没有直接把所有数据丢给 LLM，而是先用关键词做低成本过滤，再用 LLM 判断领域和内容质量。分析时会先从 ChromaDB 检索历史相似情报，同时结合 BM25 做关键词匹配，再把历史上下文和当前条目一起交给 LLM 评分。评分维度包括商业潜力、落地难度、性能指标和业务兼容性，最后得到推荐等级。
>
> 数据层上，我设计了 raw_items、analyzed_items、daily_reports、execution_logs 四张表，分别保存原始数据、分析结果、日报和执行日志。前端用 Vue 3 和 Naive UI 做控制台，支持查看今日日报、历史检索和手动触发任务。
>
> 这个项目最大的收获是，我把 LLM 能力放进了一个完整工程流程里，而不是只做单次问答。里面涉及异步采集、Agent 编排、RAG、结构化输出、任务调度、数据库建模和前端展示。

## 14. 高频面试问答

### Q1：这个项目最难的地方是什么？

答：

> 最难的是把不稳定的外部信息和不稳定的 LLM 输出接成一个稳定流程。外部信源可能失败，LLM 可能输出不合法 JSON，所以我在采集层做了单源异常隔离，在 LLM 输出层做了 JSON 提取和解析失败降级。后续如果生产化，我会继续加强重试、schema 校验和节点级日志。

### Q2：为什么要设计四维评分？

答：

> 因为技术情报不能只看热度。一个项目 star 很高，不代表适合团队落地；一篇论文性能强，也不代表商业价值高。所以我拆成商业潜力、落地难度、性能指标、业务兼容性四个维度，让模型输出更结构化，也方便排序和解释。

### Q3：RAG 的数据从哪里来？

答：

> 每次 Analyst 分析完条目后，会把标题、摘要、亮点、分数等信息写入 ChromaDB。后续分析新条目时，就可以用当前标题和描述去检索历史相似情报。

### Q4：为什么要同时用 ChromaDB 和 BM25？

答：

> ChromaDB 的向量检索擅长语义相似，BM25 擅长关键词匹配。技术情报里有很多模型名、框架名和项目名，关键词命中很重要，所以混合检索比单纯向量检索更稳。

### Q5：如果今天没有高价值情报怎么办？

答：

> Editor 会根据分析结果生成日报，高分为空也可以正常生成概览和低分列表。后续可以增加条件逻辑，比如没有强烈推荐时飞书只推送简短摘要，或者降低推送频率。

### Q6：怎么控制 LLM 成本？

答：

> 首先用关键词过滤减少进入 LLM 的数据量；其次分类和去噪使用较短 prompt 和较低 max_tokens；分析阶段只对精选内容做深度评分；并发也有限制，避免过多请求。

### Q7：这个项目怎么扩展？

答：

> 可以从四个方向扩展：第一，增加更多信源，比如 Papers With Code、Reddit、技术博客；第二，把 SQLite 换成 PostgreSQL，把调度换成 Celery；第三，加强 LLM 输出 schema 校验和重试；第四，在前端增加人工审核和反馈，把用户反馈反过来优化评分策略。

### Q8：如果让你重构，会先改哪里？

答：

> 我会先补节点级执行日志，让每个 Agent 的开始、成功、失败都能被前端准确看到；然后统一 run_id，保证日报、日志、状态查询能完整关联；接着增强 LLM 结构化输出和飞书文档写入。
